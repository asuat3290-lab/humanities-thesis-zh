# Markdown、Word 与最终交付

## 一、保存策略

保留用户源文件，另建修订版。合并前明确每一章的正式版本，避免旧稿与新稿混用。

建议以 Markdown 作为可追踪文本源，以 DOCX 作为留存、合并和提交格式。Markdown 和 DOCX 的正文、脚注及标题必须一致。

## 二、Markdown 要求

- 使用稳定标题层级。
- 脚注采用 [^1] 与 [^1]: 格式。
- 引文与正文分开，避免手工方括号编号代替脚注。
- 不在正文保留待办标记、加工说明和临时引用。
- 检查脚注引用与定义一一对应。
- 使用 UTF-8 无 BOM，避免跨工具乱码。

## 三、DOCX 要求

- 标题层级同目录一致。
- 脚注尽量转换为真实 Word 脚注，不把脚注文本堆在章末冒充脚注。
- 目录使用 Word 目录字段，并在最终文件中更新显示结果。
- 统一正文段落、引文、脚注、标题和参考文献样式。
- 封面书名和副标题不得出现不自然断行。
- 页码、分节符、页眉页脚和前置页编号应符合当前交付阶段要求。
- 不把 Markdown 标记、脚注占位符或工具引用令牌带入 Word。

## 四、使用 Documents 工作流

涉及 DOCX 时调用 Documents skill，按其编辑和验证流程处理。用户明确要求使用 Documents 或禁止 LibreOffice 时，不得暗中改用 LibreOffice。

如本机 Microsoft Word 可用，可用 Word 更新目录字段并导出临时 PDF 进行核验。任何 OOXML 低层修改后都要重新打开、更新字段并渲染。

## 五、结构核验

检查：

1. DOCX ZIP 包可正常打开。
2. 标题数量和层级符合预期。
3. 目录字段存在，显示条目完整，不把“目录”自身收录。
4. 正文脚注引用数与脚注定义数一致。
5. 没有 [^1]、# 标题、TODO 等残留标记。
6. 封面、摘要、目录、正文、结论和参考文献齐全。
7. 文件名与版本日期正确。

可运行：

~~~powershell
python scripts/audit_thesis_docx.py thesis.docx --strict
~~~

## 六、视觉核验

最终交付前渲染全部页面并检查：

- 封面标题是否孤字换行；
- 目录是否重叠、断页异常或页码未更新；
- 外语术语和长人名是否溢出；
- 大段引文是否被截断；
- 脚注区是否拥挤、跨页错位；
- 标题是否落在页尾；
- 参考文献是否出现悬挂缩进或编号错位；
- 是否存在空白页、超大间距、字体缺失和乱码。

最终版本必须基于最新一次渲染。不能在渲染后继续修改却不复查。

## 七、交付回复

只报告：

- 已完成的修改类型；
- Markdown 和 DOCX 文件链接；
- 脚注、目录和页面核验结果；
- 仍需人工确认的少量事项。

不要把临时 PDF、渲染图片、构建脚本或内部审查文件作为最终成果交付，除非用户明确要求。
